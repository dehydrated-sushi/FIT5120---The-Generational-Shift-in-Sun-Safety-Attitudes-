01 / Overview

Why This Stack

For this project we chose React as the frontend framework and Flask (Python) as the backend, connected to a PostgreSQL database and the OpenWeatherMap API for live UV data.

This combination was selected because it allows us to build a fast, interactive web app that can handle real-time data, dynamic visualisations, and user-specific inputs like location and skin tone, without overcomplicating the architecture for a three-week build.

02 / Frontend

React

What it is

React is a JavaScript library for building user interfaces through reusable components.

Why we chose it

Our app has multiple interactive features (UV alerts, visualisations, clothing recommendations) that each need to update dynamically based on user input and live data. React handles this cleanly without reloading the page, making the experience feel fast and responsive for our target audience of young Australians.

03 / Backend

Flask (Python)

What it is

Flask is a lightweight Python web framework used to build APIs and server-side logic.

Why we chose it

Our team has stronger Python experience than JavaScript backend, and Flask is simple to set up for a project of this scope. It handles API calls to OpenWeatherMap, processes UV data, queries the PostgreSQL database via SQLAlchemy, and serves responses to the frontend without the overhead of a heavier framework.

04 / Decision Rationale

Why These
Over the Alternatives

Chosen / Frontend

React

  • Dynamic content (UV levels, charts, clothing recommendations) updates based on user location without extra JS complexity.
  • Component-based structure lets each feature be built and tested independently, suiting parallel teamwork.
  • Large ecosystem of libraries like Recharts plug directly in, saving build time.
  • Produces a single-page app experience natural for a mobile-first young audience.
Not chosen / Frontend

Plain HTML / CSS

  • Cannot handle dynamic content without significant extra JavaScript complexity.
  • No built-in component model makes parallel development harder.
  • Lacks access to modern visualisation libraries without significant wiring.
  • Would not produce a natural single-page app experience for mobile users.
Chosen / Backend

Flask

  • Team is more proficient in Python, reducing onboarding time during a three-week sprint.
  • Lightweight with minimal setup, suited to the scope of this iteration.
  • Python has strong data handling libraries for processing AIHW cancer datasets.
  • Integrates cleanly with PostgreSQL via SQLAlchemy and external APIs without unnecessary complexity.
Not chosen / Backend

Node.js

  • Team would require additional ramp-up time on JavaScript backend patterns.
  • More configuration overhead for a small-scope three-week project.
  • Python data libraries like pandas are not available in the Node ecosystem.
  • No meaningful performance advantage for this scale of application.
05 / Other Tools

Supporting
Technologies

PostgreSQL
Database

Relational database used to store and retrieve static datasets including AIHW cancer incidence and mortality data, and location reference data. Chosen for its reliability, structured query support, and compatibility with Flask via SQLAlchemy.

OpenWeatherMap
External API

Provides real-time UV index data based on the user's location, used to power UV alerts and clothing recommendations.

Recharts
Visualisation

A React-compatible charting library used to render skin cancer incidence and UV trend visualisations cleanly within the frontend.

Vercel
Hosting

Hosts the React frontend, providing fast deployment and a live public URL for mentor review and user testing.

Framer Motion
Animation — TBC

A React animation library for page transitions and UI animations. Final decision on usage to be confirmed during development.

AOS
Scroll Animation — TBC

Lightweight scroll animation library. Being considered as an alternative to Framer Motion depending on implementation needs.
